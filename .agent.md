## 🏗️ Architecture Principles

### Core Design Patterns

#### 1. **Clean Architecture Layers**
```
┌─────────────────────────────────────────┐
│  Presentation / CLI / APIs              │ (User Interface)
├─────────────────────────────────────────┤
│  Use Cases / Strategy Managers          │ (Application Logic)
├─────────────────────────────────────────┤
│  Entities / Models / Domain Logic       │ (Business Rules)
├─────────────────────────────────────────┤
│  Frameworks & Drivers (DB, Broker)      │ (Technical Details)
└─────────────────────────────────────────┘
```

#### 2. **Dependency Direction**
- Inner layers (domain models) know nothing about outer layers
- All dependencies point inward
- Strategy code doesn't know about database implementation
- Broker API changes don't cascade to strategy logic

#### 3. **Event-Driven Over Polling**
- Replace polling loops with event buses
- Decouple producers (data sources) from consumers (strategies)
- Enable realtime + backtesting on same event stream
- Example events: CANDLE_CLOSED, SIGNAL_GENERATED, ORDER_FILLED

#### 4. **Type Safety**
- Use dataclasses for all domain models
- Enum for market regimes, signal types, order statuses
- TypedDict for configuration structures
- Avoid dict[str, Any] - explicit is better than implicit

#### 5. **Strategy Pattern with Regime Selection**
- Strategy interface with common contract
- RegimeManager determines which strategy to use
- StrategyManager orchestrates strategy execution
- No mixing of strategy logic - single responsibility

## 💾 Database & Data Architecture

The agent recommends:

### Primary Choice: PostgreSQL + TimescaleDB
**Why**:
- ACID transactions (critical for trading)
- Time-series optimization (TimescaleDB extension)
- Partitioned storage (reduce query time)
- Replication/HA available
- JSON support for flexible config storage

### Alternative: ClickHouse
**Why**: 
- Exceptional OLAP performance
- Compressed storage (cost-effective at scale)
- Distributed query execution
- Lower operational complexity
- Better for analytics/backtesting

### Caching: Redis
- Real-time position tracking
- Strategy state cache
- Tick data buffer before disk
- Session management

### Essential Table Schemas
```sql
-- Candlestick data (time-partitioned)
CREATE TABLE candles (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(20),
    timeframe VARCHAR(5),  -- 1m, 5m, 1h, 1d
    timestamp TIMESTAMPTZ NOT NULL,
    open DECIMAL(20,6),
    high DECIMAL(20,6),
    low DECIMAL(20,6),
    close DECIMAL(20,6),
    volume BIGINT,
    UNIQUE(symbol, timeframe, timestamp)
);
SELECT create_hypertable('candles', 'timestamp', if_not_exists => TRUE);
CREATE INDEX idx_candles_symbol_time ON candles(symbol, timestamp DESC);

-- Tick data (high frequency)
CREATE TABLE ticks (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(20),
    timestamp TIMESTAMPTZ NOT NULL,
    bid DECIMAL(20,6),
    ask DECIMAL(20,6),
    bid_qty BIGINT,
    ask_qty BIGINT,
    last_trade_price DECIMAL(20,6),
    last_trade_qty BIGINT
);
SELECT create_hypertable('ticks', 'timestamp', if_not_exists => TRUE);

-- Strategy signals
CREATE TABLE signals (
    id BIGSERIAL PRIMARY KEY,
    strategy_name VARCHAR(100),
    symbol VARCHAR(20),
    timestamp TIMESTAMPTZ NOT NULL,
    signal_type VARCHAR(20),  -- BUY, SELL, HOLD
    confidence DECIMAL(3,2),  -- 0.0-1.0
    price DECIMAL(20,6),
    regime VARCHAR(20),
    indicator_values JSONB
);

-- Trades (execution)
CREATE TABLE trades (
    id BIGSERIAL PRIMARY KEY,
    signal_id BIGINT REFERENCES signals(id),
    symbol VARCHAR(20),
    entry_time TIMESTAMPTZ,
    entry_price DECIMAL(20,6),
    stop_loss DECIMAL(20,6),
    target DECIMAL(20,6),
    quantity INT,
    exit_time TIMESTAMPTZ,
    exit_price DECIMAL(20,6),
    pnl DECIMAL(20,6),
    status VARCHAR(20)  -- OPEN, CLOSED, CANCELLED
);

-- Positions (current state)
CREATE TABLE positions (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(20),
    quantity INT,
    entry_price DECIMAL(20,6),
    current_price DECIMAL(20,6),
    pnl DECIMAL(20,6),
    updated_at TIMESTAMPTZ,
    UNIQUE(symbol)
);

-- Strategy state (persistence)
CREATE TABLE strategy_state (
    id BIGSERIAL PRIMARY KEY,
    strategy_name VARCHAR(100),
    symbol VARCHAR(20),
    state JSONB,  -- Indicator values, last signal, etc.
    updated_at TIMESTAMPTZ,
    UNIQUE(strategy_name, symbol)
);
```

## 🔄 Event-Driven Architecture

### Core Events
```python
@dataclass
class Event:
    timestamp: datetime
    symbol: str
    
@dataclass
class CandleClosedEvent(Event):
    ohlc: dict
    volume: int

@dataclass
class SignalGeneratedEvent(Event):
    strategy: str
    signal_type: str  # BUY, SELL, HOLD
    confidence: float
    reason: str

@dataclass
class OrderFilledEvent(Event):
    order_id: str
    price: float
    quantity: int

@dataclass
class PositionUpdatedEvent(Event):
    position: Position
    unrealized_pnl: float
```

### Event Bus Pattern
```python
class EventBus:
    def subscribe(self, event_type: type[Event], handler: Callable) -> None: ...
    def publish(self, event: Event) -> None: ...
    def unsubscribe(self, event_type: type[Event], handler: Callable) -> None: ...
```

### Strategy as Event Consumer
Strategies subscribe to CANDLE_CLOSED events, not polling for data.

## Realtime Monitor Runbook — Realtime monitor & cautious order placement

### Purpose
- Run the trading system in realtime when market is live.
- Monitor logs, detect and fix runtime issues where safe, and iterate until the project is stable.
- Default to `dry-run` for order actions; require explicit operator confirmation for any real orders.

### Operation modes
- `monitor` (default): observe-only, tail logs, collect diagnostics, no simulated orders.
- `dry-run`: run strategies and simulate order placement (recommended for continuous validation).
- `live`: allow real orders only with explicit confirmation (see Safety below).

### How to run
- Monitor:
    - `python tradingsystem/main_refactored.py --agent-mode monitor`
- Dry-run:
    - `python tradingsystem/main_refactored.py --agent-mode dry-run`
- Live (explicit confirmation required):
    - Method A (sentinel file): create the sentinel file `GO_LIVE` in workspace root and run with `--agent-mode live --confirm-file ./GO_LIVE`.
    - Method B (approve tool): run `python tools/approve_orders.py --session <id>` and then `python tradingsystem/main_refactored.py --agent-mode live --agent-approve`.

### Safety & Confirmation Rules
- Never place live orders unless one of the explicit confirmations is present.
- Before the first live order in a session require:
    1. Presence of `GO_LIVE` and a short human message in `logs/GO_LIVE_CONFIRM.txt`; OR
    2. A successful `tools/approve_orders.py` invocation recorded in `logs/`.
- If `GO_LIVE` is removed or `CONFIRM_ORDERS` env var is unset, switch to `dry-run` and stop placing orders.

### Log Monitoring
- Primary logs live under the `logs/` directory (subfolders by date).
- Quick tail (PowerShell):
    - `Get-Content .\\logs\\<date>\\*.log -Wait -Tail 50`
- Optionally: scaffold `tools/log_monitor.py` to parse errors and publish summaries to `logs/issues/`.

### Auto-diagnostics & Safe Fixes
- On recoverable exceptions:
    - Save stack trace + last 200 log lines to `logs/last_error_<ts>.log`.
    - Attempt a single safe recovery (re-init connection or restart component) up to N attempts (default 3).
    - If repeated failures occur, stop auto-restarts and escalate to operator.
- On dangerous conditions (balance drift, unexpected fills, unauthorized account changes):
    - Immediately switch to `monitor`, stop order attempts, and create `logs/ALERT_<ts>.md`.

### Issue Tracking & Code Changes
- Create minimal reproducible issue files under `logs/issues/` for non-auto-fixable errors.
- Suggest code fixes or `pip install` steps; do not apply code patches that affect order placement without explicit operator approval.

### Minimal Recommended Additions
- `tools/log_monitor.py` — parse and summarize errors, create issues.
- `tools/approve_orders.py` — safe approval flow that writes/verifies `GO_LIVE` sentinel.
- `run_supervised.ps1` — Windows wrapper to supervise `main_refactored.py`.

### Telemetry & Heartbeat
- Agent must update `heartbeat.json` every 30s with state `{ "mode": "...", "last_error": null, "uptime_s": 0 }`.

### Operator Guidelines
- Start in `monitor` for at least one market session.
- Move to `dry-run` to validate simulated order flows.
- Only enable `live` when present and having confirmed sentinel plus recent log review.
